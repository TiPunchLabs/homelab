# Migration réseau 192.168.10.0/24 — Plan d'exécution

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Basculer le homelab (Proxmox + 8 VMs/LXCs + IaC) du subnet `192.168.1.0/24` vers `192.168.10.0/24` sans destroy/recreate, en préservant tous les volumes et états applicatifs.

**Architecture:** In-place network reconfiguration. IaC (Terraform/Ansible) mis à jour AVANT la bascule physique. Bascule manuelle via console Proxmox (VMs) ou `pct set` (LXCs). Re-convergence Ansible une fois les hôtes joignables. Commit unique en fin de migration sur branche dédiée.

**Tech Stack:** Proxmox VE, netplan (Ubuntu 24.04), Terraform (bpg/proxmox), Ansible, Caddy, Pi-hole, WireGuard, Kubernetes (kubeadm).

**Spec:** [`docs/superpowers/specs/2026-04-28-network-migration-192-168-10-design.md`](../specs/2026-04-28-network-migration-192-168-10-design.md)

**État au démarrage du plan :**
- Branche active : `chore/network-migration-192-168-10`
- WIP Komodo : stashed sous `wip-komodo-pre-network-migration`
- Pool DHCP box : déjà reconfiguré sur `192.168.10.200-250`
- pve : déjà accessible en `192.168.10.100`
- VMs running : `dockhost-90`, `bastion-60`, `vpngate-50` (confirmé via `qm list`)
- VMs stopped : `kubebeast-40`, `kubebeast-41`, `kubebeast-42` (à démarrer en Phase 2)

## Phase 2 — Bascule réseau

> ⚠️ **Pour chaque VM**, créer un snapshot Proxmox AVANT modification (point de rollback). Pour les LXCs, le snapshot est moins critique car la modif est triviale et reversible via `pct set`.

### Task 2.1 : Bascule `dockhost-90`

**Cible :** VM 9090, IP cible `192.168.10.90/24`

- [ ] **Step 1 : Créer un snapshot**

```bash
ssh pve "sudo qm snapshot 9090 pre-network-migration --description 'Avant migration vers 192.168.10.0/24'"
```

Expected : `snapshotting 'drive-...' (...)` puis pas d'erreur.

- [ ] **Step 2 : Ouvrir la console noVNC**

Dans un navigateur : `https://192.168.10.100:8006` → datacenter → proxmox → 9090 (dockhost-90) → bouton **Console**.

Se logger avec l'utilisateur `ansible` (ou `root` si configuré).

- [ ] **Step 3 : Désactiver l'override cloud-init**

Dans la console :

```bash
echo 'network: {config: disabled}' | sudo tee /etc/cloud/cloud.cfg.d/99-disable-network-config.cfg
```

Expected : la commande affiche le contenu écrit.

- [ ] **Step 4 : Identifier le nom de l'interface réseau**

```bash
ip link | grep -v lo:
```

Noter le nom (probablement `eth0`, parfois `ens18` ou `ens192`). À utiliser dans la suite.

- [ ] **Step 5 : Éditer netplan**

```bash
sudo nano /etc/netplan/50-cloud-init.yaml
```

Remplacer le contenu par (en remplaçant `eth0` par le nom réel relevé en step 4) :

```yaml
network:
  version: 2
  ethernets:
    eth0:
      addresses: [192.168.10.90/24]
      routes:
        - to: default
          via: 192.168.10.1
      nameservers:
        addresses: [192.168.10.1]
```

Sauvegarder (Ctrl+O, Entrée, Ctrl+X).

- [ ] **Step 6 : Sécuriser les permissions du fichier (avertissement netplan)**

```bash
sudo chmod 600 /etc/netplan/50-cloud-init.yaml
```

- [ ] **Step 7 : Appliquer**

```bash
sudo netplan apply
```

Expected : aucun output (silencieux). Si warnings sur permissions → vérifier step 6.

- [ ] **Step 8 : Vérifier la nouvelle config**

```bash
ip -4 addr show | grep inet
ip route | grep default
ping -c 2 192.168.10.1
```

Expected :
- `inet 192.168.10.90/24`
- `default via 192.168.10.1 dev eth0`
- ping OK

- [ ] **Step 9 : Tester depuis le laptop**

```bash
ssh-keygen -R 192.168.1.90 2>/dev/null
ssh-keygen -R dockhost-90 2>/dev/null
ssh dockhost-90 "hostname && ip -4 addr show | grep inet"
```

Expected : connexion OK, IP `192.168.10.90` confirmée.

- [ ] **Step 10 : Vérifier que Docker est toujours up**

```bash
ssh dockhost-90 "docker ps --format 'table {{.Names}}\t{{.Status}}'"
```

Expected : tous les conteneurs `Up`. Si `Restarting` → analyser cas par cas (peut être un service qui dépendait du DNS — sera réglé en Phase 3).

## Rollback (en cas de catastrophe)

Pour une VM/LXC qui ne revient pas après bascule :

```bash
# VM
ssh pve "sudo qm rollback <vmid> pre-network-migration"

# LXC
ssh pve "sudo pct rollback <ctid> pre-network-migration"
```

L'état revient au pré-migration (mais l'IP redevient `192.168.1.X` — donc l'hôte sera à nouveau injoignable depuis le nouveau réseau, à reprendre via console).

Pour annuler les changements IaC en local :

```bash
git checkout main
git branch -D chore/network-migration-192-168-10
git stash pop  # récupérer le WIP Komodo si besoin
```
